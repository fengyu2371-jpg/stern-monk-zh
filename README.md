# stern-monk-zh-tw v10｜修士功能面板版

禊月堂繁體中文「修士」Discord Bot。

## 回答流程

1. 先查 `data/faq_zh_tw.json`（不使用 API）
2. FAQ 無命中時查 `data/tutorials_zh_tw.json`（不使用 API）
3. 兩份固定知識庫都無命中時，固定回覆「目前沒有正式資料」
4. 教學與規則查詢沒有 OpenAI 呼叫路徑，正式知識庫不會送往 API
5. 只有主面板的「告解」與「本週神諭」會使用 OpenAI API

## 目前功能

一般玩家不需要記憶大量斜線指令，所有功能都集中在固定主面板：

```text
🎓 學生資料　🏘️ 城下町　📖 神諭冊
📚 教學　　　🕯️ 告解
```

公開斜線指令只保留兩個管理用途：

- `/建立修士面板`：管理員建立固定功能面板
- `/修士狀態`：管理員檢查服務狀態

面板內可完成：

- 入學登記、查看與修改學籍、神諭偏好、刪除學籍
- 查看商店街與校外居住地、管理與登記個人地點
- 建立本週神諭、翻閱神諭冊、標記已完成／未完成
- 選擇正式教學主題或輸入問題查詢本地 FAQ
- 開啟私密告解表單

修士不代理神父 Bot 的正式遊戲操作，也不修改罪惡值、學分、體力、背包或玩家資料。

## 本地知識庫

- `data/tutorials_zh_tw.json`：正式教學、關鍵字、規則、注意事項與修士主題台詞
- `data/faq_zh_tw.json`：固定 FAQ 問法與正式回答
- `knowledge.py`：JSON 驗證、FAQ 優先配對、教學配對及未知問題固定回覆
- `persona.py`：互動界線與情緒低落偵測
- `PERSONA.md`：修士正式人格規範

啟動時若 JSON 缺檔、格式錯誤、欄位不完整、教學 ID 重複，或 FAQ 引用了不存在的教學，程式會直接停止並顯示資料錯誤，不會帶著不完整規則上線。

## Railway 環境變數

```env
MONK_TOKEN=修士的 Discord Bot Token
GUILD_ID=禊月堂的 Discord 伺服器 ID
MONK_CHANNEL_ID=修士唯一允許回覆的 Discord 頻道 ID

AI_ENABLED=true
AI_CONFESSION_ENABLED=true
AI_ORACLE_ENABLED=true
OPENAI_API_KEY=你的 OpenAI API Key
OPENAI_MODEL=gpt-5.6-terra
AI_DAILY_LIMIT=1
AI_MAX_OUTPUT_TOKENS=800
ORACLE_MAX_OUTPUT_TOKENS=700
MONK_DB_PATH=/app/storage/monk.db
BOT_LANGUAGE=zh-TW
```

`AI_DAILY_LIMIT=1` 代表不限制；目前僅限管理員測試時可先維持此設定，正式開放後再決定是否加上限。

`AI_ENABLED` 是 AI 總開關，`AI_CONFESSION_ENABLED` 控制告解，`AI_ORACLE_ENABLED` 控制每週神諭。教學與規則查詢沒有 OpenAI 呼叫路徑。

`MONK_CHANNEL_ID` 是必要設定。修士的所有斜線指令只能在這個頻道執行；其他頻道只會收到私人導引訊息，不會查詢知識庫或呼叫 OpenAI。請在 Discord 開發者模式中複製目標頻道 ID，再加入 Railway Variables。

所有 Railway 變數會由 `config.py` 集中解析。布林值或數字格式錯誤時程式會停止啟動，避免帶著錯誤設定上線；Token 與 API Key 不會寫入紀錄。

目前每日告解次數使用記憶體計算，Railway 重新部署或重啟後會歸零。教學問題查詢不會呼叫 AI；神諭則以每位學生每週一頁限制用量。

告解使用獨立提示詞，不會取得遊戲知識庫，也不能回答遊戲數值或規則。告解回覆在 Discord 中僅指令使用者可見；送往 Responses API 時設定 `store=True`，因此玩家告解輸入與修士輸出會保存並顯示在所屬 OpenAI Project 的 Responses Logs。API 無法使用或今日次數已滿時，會改用原本的本地告解回覆。

AI 告解會收到玩家顯示名稱、告解內容、目前模式與正式罪惡值結果四個欄位，並以「已聽見告解 → 整理重點與下一步 → 平穩結語」三段式回覆。動態欄位會先跳脫處理，回覆保留段落並限制在最多 220 字。

告解正文以禊月堂世界觀為準：修士確實位於教堂告解室內主持告解，不會把告解解讀成比喻、模擬或一般客服對話。試行版與罪惡值限制只顯示在 Discord 回覆下方，不混入 AI 的告解正文。

告解提示內含「偷喝朋友飲料」「忘記上課三次」「藉告解告白」三組管理員核定範例。一般告解採三段中文引號；戀愛越界採兩段拒絕，並在本地直接處理、不送往 OpenAI API。

使用 `gpt-5-nano` 時，程式會固定採用 `minimal` 推理強度，避免短回覆的 token 額度全部消耗在推理而沒有可見文字。若 API 仍回傳空內容，Railway 紀錄只會顯示回應狀態、未完成原因及 token 統計；告解輸入與模型輸出則依 `store=True` 保存於 OpenAI Responses Logs。

## 測試

測試只使用暫存 JSON 與純記憶體假 AI，不啟動 Discord、不呼叫 OpenAI，也不連接任何資料庫：

```bash
python -m unittest discover -s tests -v
```


## 回覆保證

- 固定 FAQ 永遠優先於關鍵字教學，兩者都不使用 API。
- 回覆數值與規則只取自知識庫，不由修士或 AI 自行補寫。
- 教學與規則查詢永遠不呼叫 OpenAI；未知問題一律回覆「目前沒有正式資料」。
- 修士不讀寫 `church.db`，也不修改玩家數值、背包、身分組或活動狀態。
- 戀愛、曖昧、調情、成人內容與配對要求會在本地直接拒絕，不會送往 API。
- 問題含焦慮、自責或情緒低落語句時，回答會停用吐槽並改用平穩提醒。
- AI 告解只提供一次性陪伴，不會寫入玩家資料，也不能處理正式罪惡值。
- 一般問答使用精簡知識庫內容；目前最長固定回覆仍控制在約 200 字內。

## v8 學籍、學院街區與神諭冊

### Railway Volume

修士現在會保存玩家學籍、地點與神諭頁面，因此必須建立自己的 Railway Volume。

**掛載位置請使用：**

```text
/app/storage
```

**不要掛到 `/app/data`。**  
專案原本的教學 JSON 就放在 `/app/data` 對應的程式資料夾中；若把 Volume 掛到該位置，會遮住 `tutorials_zh_tw.json` 與 `faq_zh_tw.json`，導致修士無法啟動。

Variables：

```env
MONK_DB_PATH=/app/storage/monk.db
AI_ORACLE_ENABLED=true
ORACLE_MAX_OUTPUT_TOKENS=700
```

### 週次規則

神諭冊畫面以月份內每七天為一週：

- `7-1`：7 月 1 日～7 日
- `7-2`：7 月 8 日～14 日
- `7-3`：7 月 15 日～21 日
- `7-4`：7 月 22 日～28 日
- `7-5`：7 月 29 日～月底

資料庫內部會保存完整年份，例如 `2026-07-4`，避免不同年份互相覆蓋。

### 神諭限制

- 每名學生每週只建立一頁，不提供一般玩家無限重抽。
- 玩家姓名只用於稱呼，不會被拿來推測創作主題。
- 若填有固定同行者，神諭會以學生與同行者兩人為核心。
- 商店、住處與工作室只有在玩家允許時才會被神諭採用。
- 神諭頁面可以切換上一頁／下一頁，並標記「已完成」或「未完成」。
- 互動按鈕在訊息開啟後可使用約 15 分鐘；失效後重新執行 `/神諭冊` 即可。


## v9 學生資料中心面板

管理員在指定的修士頻道執行一次：

```text
/建立學生資料面板
```

修士會建立一張公開固定面板，包含：

```text
📚 查看我的學生資料
```

任何學生按下後，只會收到自己的私密資料，其他人無法看到。

私密資料選單包含：

- 學籍總覽
- 神諭偏好
- 我的地點
- 開啟神諭冊

面板採用 Discord Persistent View：

- `timeout=None`
- 固定 `custom_id`
- Bot 啟動時使用 `client.add_view(...)` 重新註冊

因此 Railway 重啟或重新部署後，舊面板上的按鈕仍可繼續使用。

## v11 神諭成本與地點公開設定

### 精簡神諭輸入

每週神諭不再傳送：

- 完整 JSON
- 內部週次與完整日期
- 入學年份
- 學院
- 個人簡介
- 空白欄位
- 多個地點完整資料

現在只會傳送有內容的必要欄位：

- 學生稱呼
- 固定同行者
- 主修
- 喜歡／避免題材
- 最多 3 個本週關鍵字
- 最多 1 個可用地點，簡介最多約 80 字

週次仍保存在 `monk.db`，但不送進 API。

### 編輯地點公開狀態

從修士主面板進入：

```text
🏘️ 城下町
→ 👁️ 公開設定
→ 選擇自己的地點
→ 切換「公開顯示：是／否」
```

設為不公開後：

- 不會出現在公開商店街
- 不會出現在公開校外居住地
- 仍保存在玩家自己的地點資料中
- 是否可供神諭使用仍依原本「神諭可用」設定判斷


## v12 地點公開與神諭用途分離

玩家登記的所有地點，都可以成為該玩家每週神諭的素材。

- 公開：會出現在城下町公開名單，也可進自己的神諭。
- 不公開：不會出現在公開名單，但仍可進自己的神諭。
- 不再提供「神諭可用：是／否」選項。
- Bot 啟動時會把舊資料中的地點統一設為可作神諭素材。
- 每次生成仍只挑最多 1 個地點送入 API，不會一次送出全部地點。

「公開／不公開」現在只負責城下町展示，不會影響神諭。


## v13 不限次數神諭測試

目前為管理員測試階段，已暫時移除「每位玩家每週只能建立一頁」限制。

從主面板進入：

```text
📖 神諭冊
→ ✨ 抽取新神諭
```

每按一次都會：

- 呼叫一次 OpenAI API
- 建立一頁新的神諭
- 保存至 `monk.db`
- 同一個 `7-4` 可以存在多頁

每次測試抽取會使用不同的內部抽次編號挑選關鍵字與地點，避免同週一直選到完全相同的素材。

### 刪除神諭

神諭冊新增：

```text
🗑️ 刪除此頁
```

按下後會先顯示二次確認。確認後只刪除目前頁面，其他神諭不受影響。

### 舊資料庫自動升級

v12 以前的 `oracle_pages` 有每週唯一限制。Bot 啟動時會自動重建該資料表並保留既有神諭，不需要手動刪除 `monk.db`。


## v14 使用次數限制

Railway Variables：

```env
AI_DAILY_LIMIT=1
ORACLE_WEEKLY_LIMIT=3
```

### 告解

- 每位玩家每天最多 1 次 AI 告解。
- 日期依臺灣時間重置。
- 次數保存在 `monk.db`，Railway 重啟或重新部署不會歸零。
- 大猩猩外號、告白、調情等由本地規則攔截的內容不呼叫 API，也不扣次數。
- API 呼叫失敗會自動退還該次名額。

### 神諭

- 每位玩家每個月週次（例如 `7-1`）最多抽取 3 次。
- 次數保存在 `monk.db`。
- API 生成失敗會退還名額。
- 刪除神諭頁面不會退還抽取次數，避免藉由刪除無限重抽。
- 升級前已存在的神諭頁面會自動計入對應週次的已用次數。

### 按鈕所有權

- 公開主面板可以由所有玩家按下，但只會開啟按鈕操作者自己的資料。
- 學籍、地點管理、神諭冊、刪除確認、教學等個人子頁面都綁定原操作者。
- 其他玩家操作他人的個人頁面時，只會收到私密拒絕提示。

## v15 公開資料與地點操作流程

### 我的地點

「學生資料 → 我的地點」與「城下町 → 我的地點」現在都會進入同一張管理頁，直接提供：

- 新增地點
- 公開設定
- 查看全部（私密）
- 重新整理
- 返回上一層

玩家不需要先理解「新增功能藏在城下町」的結構。

公開頁只列出該玩家的公開地點。不公開地點只顯示數量，不會公開名稱、區域或簡介。

### 公開與私密範圍

公開顯示：

- 學生學籍與個人簡介
- 公開的商店與校外住處
- 神諭冊內容
- 遊戲教學

維持私密：

- 入學與修改表單
- 神諭偏好詳細內容
- 不公開地點資料
- 刪除確認
- 告解內容與回覆

所有玩家專屬控制介面仍使用 `owner_id` 驗證。其他玩家可以閱讀公開內容，但無法代替原玩家修改、標記、刪除或操作個人地點。


## v16 成員開放與私密控制台

Discord 無法讓同一則公開訊息上的按鈕「只對某個人可點，對其他人不可點」。  
因此 v16 改成：公開主面板所有成員都能按，但按下後開啟的是該成員自己的私密控制台。

### 成員都能玩

主面板上的功能：

- 學生資料
- 城下町
- 神諭冊
- 教學
- 告解

都開放一般成員使用。

### 其他玩家不能點你的

玩家專屬操作一律只出現在操作者自己的 ephemeral 私密訊息裡：

- 修改學籍
- 我的地點
- 新增地點
- 公開設定
- 神諭冊翻頁
- 標記已完成／未完成
- 刪除神諭
- 告解

其他玩家看不到這些私密控制台，也就無法點擊或代替操作。

### 不需要寫死你的 ID

目前不需要設定你的 Discord ID。  
管理指令仍由 Discord 權限控管：

- `/建立修士面板`
- `/修士狀態`

這兩個仍需要「管理伺服器」權限。若未來真的要只允許特定帳號管理，再加 `BOT_OWNER_ID` 會比較乾淨。


## v17 公開可見、本人可操作

v17 改成公開頁面可見，但玩家專屬按鈕只允許原操作者使用。

Discord 目前無法做到「同一則公開訊息裡，按鈕對 A 可點、對 B 顯示灰色不可點」。  
因此本版採用 owner guard：

- 其他人看得到公開頁面。
- 其他人按按鈕時會收到私密拒絕提示。
- 公開訊息不會被其他人的操作改動。
- 原操作者操作時，盡量使用 `edit_message` 或 `edit_original_response` 修改同一則訊息，不再一直彈出新訊息。

仍維持私密的內容：

- 尚未入學時的入學設定入口
- 修改表單本身
- 刪除確認
- 告解內容與回覆
- 教學問題查詢 Modal 的輸入結果


## v18 學生資料頁直接新增地點

「學生資料」頁新增：

```text
➕ 新增地點
```

玩家不必先進入「我的地點」或「城下町」才能建立商店、住處與工作室。

## 店主、經營者與居住者

地點登記表單新增角色欄位：

- 商店、餐館、書店、魔藥工房、診所：店主／經營者
- 校外住處：居住者
- 工作室、社團據點與其他：負責人／使用者

欄位預設使用學生的希望稱呼，但可改成：

- 其他角色名稱
- 多名共同經營者
- 伴侶或室友
- 舊企劃原本的店主

公開地點頁會明確顯示經營者／居住者。

舊地點會在 Bot 啟動時自動以原登記學生的希望稱呼回填，不需要重新登記。
